A. Frontera y Movilidad (.NET MAUI & YARP)
1 App Móvil (.NET MAUI) — Iniciar Sesión

La app guarda el JWT en SecureStorage (cifrado del dispositivo). Esto simula el login contra un Identity Server.

💬 "La app MAUI guarda el JWT en SecureStorage — cifrado nativo del dispositivo. Al hacer cualquier petición, el AuthHandler lo adjunta automáticamente como Bearer token."
2 App Móvil — Consultar Producto (via Gateway)

La app llama al Gateway (YARP). El Gateway valida el JWT y enruta a Product.Api → Inventory.Api.

Flujo de la petición:
📱 MAUI ──JWT──▶ 🚪 Gateway (YARP) ──valida──▶ 📦 Product.Api ──HTTP──▶ 🏭 Inventory.Api
puerto 5110                 puerto 5298            puerto 5293

✅ Verificación: La app muestra {"productId":1,"stock":50,"sku":"BOLETA-MEDELLIN-VIP"}

💬 "El flujo completo es: la app MAUI adjunta el JWT automáticamente con el AuthHandler, el Gateway lo valida (issuer, audience, firma, expiración), y solo si es válido enruta la petición a Product.Api, que a su vez consulta Inventory.Api. Sin token → 401 en el Gateway, nunca llega al backend."
3 API Gateway (YARP) — Validación JWT

El Gateway valida issuer, audience, firma y expiración del token ANTES de redirigir al backend.

Esperando acción...
💬 "Sin token → 401 Unauthorized. Con token válido → 200 OK. El Gateway es la primera línea de defensa — los microservicios internos nunca reciben tráfico no autenticado."
4 Rate Limiting — Control de Multitudes

100 requests/min global por IP + 5 compras/min para evitar acaparamiento de boletas.

Esperando acción...
💬 "Máximo 5 compras por minuto por IP. Al sexto intento → HTTP 429 Too Many Requests. Esto evita que un bot acapare todas las boletas."
B. Núcleo de Microservicios (Backend)
1 Venta de Órdenes — Patrón SAGA + gRPC

Order.Api reserva stock via gRPC → procesa pago → si falla, compensa devolviendo stock.

Flujo SAGA (éxito):
🛒 Order.Api ══gRPC══▶ 🏭 Inventory (ReduceStock) ──OK──▶ 💳 Pago ──OK──▶ 📨 RabbitMQ (evento)

Flujo SAGA (fallo + compensación):
🛒 Order.Api ══gRPC══▶ 🏭 Inventory (ReduceStock) ──OK──▶ 💳 Pago ──FALLA──▶ 🏭 Inventory (ReleaseStock) ↩️
Esperando acción...

✅ Verificación: protocol: "gRPC" + inventoryLatencyMs < 15ms. Si falla el pago → "Stock compensado".

💬 "La comunicación Order↔Inventory es gRPC — protocolo binario sobre HTTP/2. La latencia es 2-11ms vs ~100ms de REST con JSON. Si el pago falla, la SAGA ejecuta la compensación automáticamente devolviendo el stock."
2 Validación Binaria (gRPC) — Demostrar latencia

El campo inventoryLatencyMs en la respuesta muestra la latencia real de gRPC medida con Stopwatch.

💬 "Fíjense en inventoryLatencyMs: 2ms. Eso es lo que tarda la comunicación binaria gRPC entre Order e Inventory. El requisito era menos de 100ms — lo cumplimos con creces."
3 Precios Dinámicos — Caché Distribuida con Redis

Product.Api consulta Redis primero (cache-aside). Si no hay dato, va a Price.Api y guarda con TTL 60s.

Esperando acción...

✅ Verificación: Primera llamada más lenta (miss). Segunda más rápida (hit desde Redis).

💬 "Patrón Cache-Aside: la primera consulta va a Price.Api y guarda en Redis. Las siguientes 59 segundos vienen de Redis sin tocar la base de datos. Con 50,000 usuarios, el 90% son cache hits."
C. Comunicación y IA (Eventos y Semántica)
1 Mensajería Asíncrona — MassTransit + RabbitMQ

Order.Api publica OrderCreatedEvent a RabbitMQ (CloudAMQP). Notification.Api lo consume en segundo plano.

Flujo evento:
🛒 Order.Api ──publish──▶ 🐰 RabbitMQ (cola) ──consume──▶ 🔔 Notification.Api ──push──▶ 📱 MAUI (SignalR)

  5. Volver a levantar Notification →

Esperando acción...

✅ Verificación: En la terminal de Notification.Api aparece "Procesando evento de RabbitMQ para orden: {id}"

💬 "El evento viaja por RabbitMQ en la nube (CloudAMQP). Si Notification.Api está caído, el mensaje queda en la cola y se procesa cuando se levante — esto es desacoplamiento temporal. El productor nunca se bloquea esperando al consumidor."
2 Tiempo Real — SignalR push a la App Móvil

Notification.Api envía "TicketReady" via WebSocket a todos los clientes MAUI conectados.

✅ Verificación: En terminal Notification.Api → "Notificación push enviada via SignalR". En la app MAUI → texto morado "ALERTA EN VIVO".

💬 "SignalR mantiene una conexión WebSocket persistente con la app. Cuando el consumer procesa el evento, el Hub envía un push en tiempo real — el usuario ve la confirmación sin refrescar."
3 Buscador Inteligente — Elasticsearch + Qdrant (IA)

Elasticsearch busca por texto exacto. Qdrant busca por significado/intención usando vectores.

Esperando acción...

✅ Verificación: "algo divertido para la familia" → primer resultado es "Zona Familiar" (sin usar esas palabras exactas).

💬 "El usuario busca por 'vibe' — no necesita saber el nombre exacto del evento. Qdrant convierte el texto en un vector y busca los más similares por distancia coseno. Esto es IA aplicada a búsqueda."
D. Infraestructura y Nube (DevOps)
1 Contenerización — Dockerfile Multi-stage

Cada servicio tiene un Dockerfile optimizado: etapa BUILD (SDK) + etapa RUNTIME (solo runtime ~100MB).

💬 "Multi-stage build: la imagen final solo tiene el runtime (~100MB), no el SDK (~700MB). Menos superficie de ataque y deploys más rápidos."
2 Orquestación — Kubernetes + HPA (Auto-escalado)

13 pods corriendo. HPA escala automáticamente de 2 a 15 réplicas según CPU.

Esperando acción...

✅ Verificación: Después de kill → pods nuevos con AGE de pocos segundos (self-healing). HPA muestra min/max réplicas.

💬 "Mato los pods — simulando una caída del servidor. Kubernetes detecta que faltan réplicas y las recrea en 4 segundos. El sistema nunca se apaga. El HPA escala hasta 15 réplicas si la CPU supera 60%."
3 Automatización — GitHub Actions CI/CD

Al hacer push a main: compila → testea → construye 6 imágenes Docker → las sube a Docker Hub.

Abrir GitHub Actions (pipeline verde)

✅ Verificación: El pipeline muestra ✓ verde. Las 6 imágenes están en Docker Hub como watabares/itm-*:latest.

💬 "El pipeline se dispara automáticamente al hacer push. Compila los 6 servicios en paralelo, corre tests, y sube las imágenes a Docker Hub. Los pods de K8s pueden hacer pull de esas imágenes para actualizar."
4 Infraestructura como Código — Terraform

Un archivo main.tf que recrea toda la infraestructura AWS: VPC, subnets, EKS, nodos con auto-scaling.

Esperando acción...
💬 "Este archivo recrea TODA la infraestructura con un comando: VPC, subnets, cluster EKS con auto-scaling de 2 a 6 nodos. Es reproducible, versionada en git, y auditable."
